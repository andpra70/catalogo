Rigenera un'applicazione web React + Vite chiamata "Catalogo Opere", pensata come editor visuale per impaginare un catalogo fotografico/editoriale stampabile. L'app deve essere molto vicina all'implementazione descritta qui sotto, sia come funzionalita sia come struttura dei dati, flussi utente, default, strumenti di editing e resa visiva.

Obiettivo generale

L'app consente di:
- gestire un archivio di opere con immagine e metadati
- comporre un catalogo pagina per pagina
- applicare layout automatici alle pagine opere
- inserire, spostare, ridimensionare e modificare manualmente immagini, didascalie e blocchi testo
- cambiare formato pagina e tema tipografico/grafico
- salvare progetto e tema in locale
- esportare JSON e PDF in vista libro

Stack tecnico

- React 18 con componenti function e hook
- Vite
- CSS tradizionale in un solo file `src/styles.css`
- `html2canvas` + `jspdf` per export PDF
- `localStorage` per stato, progetti e temi
- `IndexedDB` per immagini, DB `catalogo-opere-assets`, store `images`
- nessun router
- un solo componente principale enorme `src/App.jsx` che contiene quasi tutta la logica e molti sottocomponenti UI

File principali

- `index.html`: carica Google Fonts con una lista ampia di font editoriali
- `src/main.jsx`: mount dell'app React
- `src/App.jsx`: costanti, utility, stato, logica editoriale, drag/resize, persistenza, export, componenti UI
- `src/styles.css`: interfaccia completa, pannelli, toolbar, page canvas, modali, print CSS

Prefazione curatoriale

- deve esistere una vera pagina di prefazione/curatela come prima pagina interna del catalogo
- deve comparire prima di qualsiasi pagina opere
- deve usare i template Markdown del tema per titolo e corpo
- deve essere persistita come pagina reale del catalogo
- quando il catalogo viene rigenerato dal tool layout, questa pagina deve essere preservata oppure ricreata automaticamente nella stessa posizione

Sommario / elenco opere

Il sommario deve essere generato automaticamente come `summary-page`:
- una sola pagina
- posizionata in penultima posizione, subito prima del retrocopertina
- costituita da un solo blocco testo Markdown
- il testo parte in alto, occupa tutta l'area utile della pagina
- contiene titolo `# Elenco opere` e tutte le opere con pagina relativa
- la dimensione font deve ridursi automaticamente fino a far stare tutto il contenuto su quella sola pagina
- deve essere possibile cancellare anche questa `summary-page` generata dall'interfaccia

Esempio riga sommario:
- `1. **Titolo** — *Autore* — anno: 2024 — **pag. 7**`

Gestione layout automatici

Prevedi almeno questi preset:
- `single`
- `double`
- `quad`
- `masonry`

Ogni preset genera placements automaticamente rispettando:
- formato pagina corrente
- margini tema
- aspect ratio immagini
- bound mode
- theme defaults di bordi/caption

Comportamenti richiesti del tool layout

- "Applica layout a pagina attiva" deve ricostruire solo i placement della pagina attiva
- "Genera catalogo" deve rigenerare le pagine opere del catalogo
- deve mantenere o ricreare la pagina di prefazione/curatela come prima pagina interna
- non deve distruggere elementi esistenti in copertina o retrocopertina: deve solo normalizzarli
- non deve inserire una pagina bianca/editoriale prima della lista delle opere
- la sequenza generata deve essere:
  - copertina
  - seconda di copertina se presente
  - prefazione/curatela come prima pagina interna
  - pagine opere
  - inside back
  - sommario
  - retrocopertina

Inserimento e modifica elementi

- possono essere aggiunte dalla filmstrip alla pagina attiva

Drag, resize, snap

Sistema di editing diretto con:
- drag elementi
- resize blocchi testo, immagini e didascalie
- snap opzionale a griglia
- guide magnetiche opzionali
- snap a margini/centri/pagina e, se attivo, ad altri elementi
- se `snapToGrid` e disattivato il movimento deve essere libero, senza snap implicito
- `Alt` durante drag disabilita temporaneamente lo snap
- `boundMode: "margins"` vincola all'area interna
- `boundMode: "page"` consente anche sconfinamento nei margini ma sempre entro il foglio

Cambio formato pagina

Quando cambia il formato pagina:
- tutte le pagine devono essere trattate nello stesso modo
- nessuna categoria di pagina deve avere un comportamento speciale
- testi e placements devono essere scalati e riposizionati sul nuovo formato
- i testi devono essere riportati dentro i vincoli finali del nuovo formato
- niente automatismi che espandano i blocchi testo al massimo della pagina
- le immagini e didascalie devono essere normalizzate nei vincoli finali

Cambio margini

Quando cambiano i margini tema:
- aggiornare margini di tutte le pagine
- normalizzare tutte le pagine rispetto ai nuovi vincoli
- applicare la stessa logica anche alle pagine speciali e al sommario

Normalizzazione manuale

Il comando "Riporta elementi dentro vincoli" deve:
- funzionare sulla pagina attiva
- usare la stessa logica comune di normalizzazione
- applicarsi sia a placements sia a blocchi testo

Persistenza locale

Import/export JSON

Export PDF

Discrepanze note

Questa sezione deve essere mantenuta per distinguere tra specifica target e comportamento corrente del codice esistente.

1. Prefazione curatoriale
- specifica target: il catalogo deve avere una pagina di prefazione/curatela come prima pagina interna prima delle pagine opere
- stato attuale del codice: alcuni flussi recenti di generazione catalogo possono partire direttamente dalle pagine opere senza reinserire la prefazione
- implicazione per la rigenerazione: la prefazione va trattata come parte strutturale del catalogo

2. Eliminazione summary-page
- specifica target: anche la `summary-page` generata deve essere eliminabile dall'interfaccia
- stato attuale del codice: la rimozione reale e oggi naturale per `pages` e `specialPages`, mentre il sommario e ancora una pagina derivata rigenerata dal catalogo
- implicazione per la rigenerazione: serve un modello dati o una strategia persistita che consenta di cancellare davvero il sommario generato

3. Pulsante elimina pagina
- specifica target: il pulsante nella sezione "Pagina attiva" deve essere sempre presente e funzionare su qualsiasi pagina mostrata come attiva
- stato attuale del codice: il bottone e sempre visibile, ma la semantica dati non e ancora perfettamente uniforme per tutte le categorie di pagina
- implicazione per la rigenerazione: uniformare il concetto di pagina eliminabile

4. Prompt vs codice
- questo file `prompt.txt` descrive il target desiderato per rigenerare il programma
- quando il codice corrente differisce, la differenza va annotata qui invece di essere nascosta
- durante una futura rigenerazione, prevale la specifica di questo file rispetto alle accidentalita della codebase attuale

Deliverable desiderato

Rigenera l'app completa con codice eseguibile, non una demo minimale. Mantieni la logica editoriale e operativa il piu possibile aderente alla specifica sopra, anche se internamente scegli piccole semplificazioni. L'app finale deve comportarsi come un editor professionale di cataloghi fotografici con forte controllo manuale, layout automatico e persistenza locale.
